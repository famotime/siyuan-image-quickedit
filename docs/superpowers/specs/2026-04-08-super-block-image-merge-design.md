# 超级块图片合并设计

日期：2026-04-08

## 背景

当前插件支持单张图片右键处理，以及文档级批量图片处理，但尚未支持在思源超级块上直接执行图片拼接。目标是在超级块右键菜单中新增“图片合并”命令，将超级块中的多张图片按出现顺序、顶部对齐、横向拼接为一张新图片，并将结果插入到该超级块下方。

## 目标

- 在超级块块图标右键菜单中新增“图片合并”命令。
- 仅当目标块为超级块且内部至少包含 2 张图片时显示该命令。
- 合并时只处理超级块内部图片，按 DOM 出现顺序横向拼接，顶部对齐。
- 生成结果上传为思源资源，并在超级块后插入一个新的普通图片块。
- 原超级块内容保持不变，不替换原图，不修改其他子块。

## 非目标

- 不新增设置项。
- 不修改现有单图“图片快剪”命令集合。
- 不支持自定义排列方向、间距、背景色或输出格式。
- 不改动 README、发布素材或模板目录。

## 用户交互

### 菜单入口

- 继续监听现有 `click-blockicon` 事件。
- 若当前右键目标块的 DOM `data-type` 为 `NodeSuperBlock`，则检查其内部图片数量。
- 当图片数量大于等于 2 时，在块图标菜单中增加一个顶层菜单项：`图片合并`。
- 当图片数量小于 2 时，不显示该菜单项。

### 执行反馈

- 复用插件现有独占执行保护，避免与其他图片处理任务并发。
- 执行过程中显示阶段性提示，例如读取图片、生成拼接图、上传资源、插入结果块。
- 成功后显示成功消息，说明结果已插入到超级块下方。
- 失败时显示明确错误消息，不插入半成品块。

## 架构与模块划分

### 插件入口层

文件：`src/index.ts`

职责：

- 在 `click-blockicon` 处理中识别超级块。
- 根据超级块上下文决定是否添加“图片合并”菜单项。
- 点击后调用新的超级块合并服务。
- 统一处理独占执行、消息提示和错误展示。

### 菜单构建层

文件：`src/core/menu-items.ts`

职责：

- 新增一个简洁的菜单项构建函数，返回超级块右键菜单所需的“图片合并”项。
- 保持现有单图菜单与文档批处理菜单逻辑不变。

### 图片工作流层

文件：`src/services/image-workflow.ts`

职责：

- 从超级块元素中收集图片目标，按 DOM 顺序返回。
- 读取图片资源并创建 `ImageBitmap`。
- 根据“顶部对齐横向拼接”规则合成新图片。
- 导出为 `webp` Blob，并生成用于上传的文件名。

### Kernel API 层

文件：`src/services/kernel.ts`

职责：

- 继续复用现有 `uploadAsset` 和 `insertMarkdownAfterBlock`。
- 如有必要，补充最小的新查询辅助函数，但不改动已稳定的上传/插块接口契约。

## 数据流

1. 用户在超级块块图标打开右键菜单。
2. 插件识别当前块为 `NodeSuperBlock`。
3. 插件扫描该超级块内部的图片元素并解析为图片目标列表。
4. 若列表长度至少为 2，则渲染“图片合并”菜单项。
5. 用户点击“图片合并”后，插件进入独占执行区。
6. 工作流服务按顺序读取图片并创建位图。
7. 计算目标画布尺寸：
   - 宽度为所有图片原始宽度之和。
   - 高度为所有图片原始高度的最大值。
8. 在画布上依次以 `y = 0` 绘制各图片，`x` 按累计宽度递增。
9. 导出合并结果为 `image/webp`。
10. 上传资源至思源资产目录。
11. 构造普通图片 markdown，并插入到超级块块 ID 之后。
12. 展示成功消息。

## 输出格式

- 合并结果为单张 `webp` 图片。
- 文件名建议格式：`superblock-merge-<timestamp>.webp`。
- 插入块使用普通图片 markdown，保证结果在编辑器中表现为单独图片块。

## 错误处理

- 超级块中图片不足 2 张：菜单不显示。
- 任一图片读取失败：终止任务并报错，不上传、不插入。
- 画布创建或导出失败：终止任务并报错。
- 上传失败：终止任务并报错，不插入。
- 插入失败：报错；允许出现“资源已上传但正文未插入”的孤立资源，但不宣称成功。
- 所有 `ImageBitmap` 必须在 `finally` 中关闭，避免泄漏。

- 验证非超级块或图片不足 2 张时不会追加该菜单项。

## 风险与取舍

- 画布尺寸直接使用原始分辨率，若多张超大图片同时拼接，浏览器内存占用会升高。
  当前版本接受该取舍，因为需求强调的是直接合并，不要求额外压缩或缩放策略。
- 上传成功但插入失败时，思源资源目录中可能残留未引用图片。
  当前版本接受该取舍，不额外实现资源回收逻辑，以保持实现简单且低风险。
