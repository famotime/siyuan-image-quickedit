# 当前压缩策略

本文档描述 `siyuan-image-quickedit` 当前代码中的图片压缩实现逻辑。内容以实际实现为准，对应核心代码位于 `src/services/image-workflow.ts` 与 `src/services/palette-quantization.ts`。

## 命令目标

当前压缩命令及目标体积比例：

- `compress-75`: 目标为原图大小的 `75%`
- `compress-50`: 目标为原图大小的 `50%`
- `compress-30`: 目标为原图大小的 `30%`
- `compress-10`: 目标为原图大小的 `10%`

目标字节数计算方式：

```text
targetBytes = floor(originalBytes * targetRatio)
```

## 预处理阶段

执行压缩前，插件会先完成以下步骤：

1. 根据图片块定位目标图片。
2. 读取原图二进制数据。
3. 创建 `ImageBitmap`。
4. 解析原图元数据：
   - 原始宽高
   - 原始大小
   - MIME 类型
   - 文件格式
   - 颜色深度
5. 计算图片在文档中的显示缩放比例 `displayScale`。

说明：

- `displayScale` 仅用于“需要降分辨率时”的起始点。
- 当前逻辑不会一开始就按显示尺寸改分辨率。

## 总体压缩顺序

压缩候选按以下顺序尝试：

1. 先尝试原始分辨率 `1.0`
2. 如果原始分辨率下仍无法达到目标体积，再进入降分辨率阶段
3. 降分辨率阶段从 `displayScale` 开始，并继续递减

每个分辨率下的内部顺序固定为：

1. 先只调整 WebP 质量
2. 若仍未达到目标，再在该分辨率下尝试调色板压缩

选择规则：

- 不再使用“第一个命中即返回”
- 会收集所有满足 `bytes <= targetBytes` 的候选
- 在所有达标候选中，按观感评分选择最终结果
- 如果所有候选都未命中，返回整个搜索过程中体积最小的结果

## 分辨率尝试策略

候选缩放序列由 `buildCompressionScaleSteps()` 生成。

规则如下：

1. 第一个候选始终是 `1`
2. 第二个候选是 `displayScale`
3. 后续依次尝试：
   - `displayScale * 0.9`
   - `displayScale * 0.8`
   - `displayScale * 0.7`
   - `displayScale * 0.6`
   - `displayScale * 0.5`
   - `displayScale * 0.4`
   - `displayScale * 0.3`
   - `displayScale * 0.2`
4. 每一步都会被限制到 `0.1 ~ 1` 区间
5. 重复值会被去重

例如：

- 若 `displayScale = 0.5`，顺序为：

```text
1, 0.5, 0.45, 0.4, 0.35, 0.3, 0.25, 0.2, 0.15, 0.1
```

- 若 `displayScale = 1`，顺序为：

```text
1, 0.9, 0.8, 0.7, 0.6, 0.5, 0.4, 0.3, 0.2
```

## WebP 质量策略

每个“分辨率 + 调色板”变体下，当前不会再用固定质量阶梯做完整遍历，而是采用“高低边界 + 二分逼近 + 局部细化”的方式搜索更优质量。

当前搜索边界：

```text
最高质量: 0.98
最低质量: 0.05
```

搜索流程：

1. 先尝试 `0.98`
2. 再尝试 `0.05`
3. 如果高质量已满足体积要求，则该变体直接采用高质量结果
4. 如果低质量仍无法满足体积要求，则该变体视为无法达标
5. 如果两者之间存在可行区间，则执行二分逼近
6. 二分后再做一轮局部细化采样，尽量逼近目标体积上限

这一阶段的特点：

- 不修改颜色数量
- 目标是在当前变体下找到“满足体积约束的最高可用品质”
- 达标后不会立即结束整个压缩流程，而是继续参与最终评分

## 调色板 / 颜色数压缩策略

如果“当前分辨率 + 质量阶梯”仍达不到目标体积，就在同一分辨率下继续尝试有限调色板版本。

颜色预算顺序为：

```text
256, 128, 64, 32, 16
```

每个颜色预算下，都会再次执行同样的质量搜索流程。

也就是说，同一分辨率下的完整顺序是：

1. 原始颜色数量 + 质量搜索
2. `256` 色 + 质量搜索
3. `128` 色 + 质量搜索
4. `64` 色 + 质量搜索
5. `32` 色 + 质量搜索
6. `16` 色 + 质量搜索

## 达标候选评分策略

当前实现会对所有达标候选计算一个观感评分，再选择分值最高的结果。

评分维度及权重：

- 分辨率保留度：`55%`
- 调色板保留度：`20%`
- 编码质量：`20%`
- 体积利用率：`5%`

评分倾向：

- 更高分辨率优先
- 更少的颜色压缩优先
- 更高的 WebP 质量优先
- 在前三项接近时，优先选择更接近目标体积上限的结果

这意味着当前策略追求的是：

> 在满足目标体积前提下，尽量保住分辨率、颜色丰富度和编码质量，而不是尽快找到第一个合格结果。

## 当前量化实现方式

当前的调色板压缩不是复杂的自适应调色板算法，而是基于 RGB 通道位深分配的快速量化：

1. 根据最大颜色数 `maxColors` 计算可用总 bit 数
2. 将 bit 数轮流分配给 `R/G/B` 三个通道
3. 将每个通道映射到对应离散等级
4. 保留原始 alpha 通道，不改透明度

实现特征：

- 速度快
- 实现简单
- 有损
- 透明像素的 alpha 值保持不变
- 对视觉质量的控制较粗糙，偏工程化压缩，不是高质量图像量化器

## 输出格式

当前所有压缩结果统一输出为：

```text
WEBP
```

无论原图是 PNG、JPEG、GIF 还是其他可处理格式，压缩输出都为 `.webp` 文件。

## 进度提示

压缩过程中会输出进度消息。当前主要包括：

- `正在读取图片`
- `正在尝试 {width}×{height}`
- `正在尝试 {width}×{height} / {maxColors} 色`
- `正在上传处理结果`
- `正在插入结果块`
- `正在替换原图`

## 回退规则

如果没有任何候选能达到目标体积：

- 不报“压缩失败”
- 返回搜索过程中体积最小的那个候选

只有在完全无法生成任何候选结果时，才会抛出错误：

```text
无法生成压缩结果。
```

## 当前限制

当前策略存在以下限制：

- 调色板压缩不是自适应聚类算法，颜色分布利用率有限
- 未使用有损 PNG、AVIF、JPEG XL 等其他格式策略
- 未基于图像内容类型区分照片、插画、截图
- 未针对文字截图做专门锐化或边缘保护
- 未引入 SSIM、PSNR 等视觉质量指标
- 评分仍是启发式规则，尚未引入 SSIM、Butteraugli 等感知指标
- 质量搜索默认假设文件大小随质量参数大体单调变化，极端图像上不一定严格成立

## 当前实现摘要

一句话概括：

> 先在原始分辨率下搜索最优质量，再尝试不同调色板；若仍不够，再逐步降低分辨率并重复搜索。所有达标候选都会参与观感评分，最终选择评分最高的结果；若没有达标候选，则返回体积最小结果。
